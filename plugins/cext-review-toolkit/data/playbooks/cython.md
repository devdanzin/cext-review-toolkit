# Cython Playbook

**Generator**: Cython
**Versions covered**: 3.0, 3.1, 3.2, 3.3+ (NOT 0.29.x — Cython 3 introduced semantic changes that invalidate older guidance)
**Calibrated against**: blosc2 4.1.3 (Cython 3.2.4), uvloop, h3-py, cymem, cytoolz, msgpack-python, statsmodels (51 .pyx files; Phase 0 corpus)
**Last updated**: 2026-04-27

This playbook tells the `generated-code-mapper` agent what to put into the orientation guide for a Cython project. It also tells downstream audit agents what to filter, what to elevate, and what to look out for.

---

## 1. What Cython does

Cython compiles `.pyx` source files (Python with optional C-level type annotations) to C source files via the Cython compiler. The generated C is much larger than the input `.pyx` (typically 20-40× expansion) — most of the bulk is **Cython runtime machinery** (refcount macros, exception propagation helpers, vectorcall fast paths, memoryview adapters). User code is roughly 5% of the generated C; the other 95% is template-driven.

**The maintainer's source-of-truth is the `.pyx` file.** Bugs in maintainer logic exist there. The generated `.c` is a faithful translation; if you see a bug in `.c`, the same bug is in `.pyx` (modulo Cython compiler bugs, which are extremely rare).

Cython also accepts:
- `.pxd` files — C-level declarations (analogous to C headers; declare types, externs, function signatures)
- `.pxi` files — `include`-d source files
- `# cython:` directives at the top of `.pyx` (per-file compiler options)

## 2. Source-of-truth rule

**Always cross-reference findings to `.pyx` line numbers, not `.c` line numbers.** Downstream agents reading audit reports want to fix the `.pyx`; the `.c` is a build artefact.

## 3. Cross-reference recipe (.c → .pyx)

Cython 3.x emits `/* "<filename>.pyx":<line> */` comments throughout the generated `.c`. Recipe to find the `.pyx` source for any `.c` line `N`:

```bash
# Find the most recent /* "filename.pyx":<line> */ comment before line N
sed -n '1,Np' generated.c | grep -E '/\* "[^"]+\.pyx":[0-9]+ \*/' | tail -1
```

Or programmatically: walk backward from line `N` until you hit a comment of the form `/* "<filename>.pyx":<line> */`. The last such comment indicates the `.pyx` line `N` was emitted from.

**Note**: comments are emitted at *block* granularity, not per-statement. A single `.pyx` line typically expands to 5-50 lines of `.c`; the comment marks the start of each expansion.

## 4. Generator detection

Look for these signals in priority order:

1. **`/* Generated by Cython X.Y.Z */`** at the top of any `.c` file in the build tree. Authoritative version.
2. **`cython>=X.Y` in `pyproject.toml`'s `[build-system].requires`**. Indicates the minimum version; the *actual* version is whatever pip resolved.
3. **`# cython:` directives at the top of `.pyx`**. Indicates Cython is in use; doesn't pin version.
4. **`.pyx` / `.pxd` / `.pxi` files anywhere in `src/`**. Final fallback.

If you can't determine the version from #1 or #2, default to **3.2** as a working assumption and note the uncertainty in the orientation guide.

## 5. Idioms — ACCEPTABLE noise (downstream agents should filter)

Cython runtime emits these patterns; they are NOT bugs. Each entry includes a grep regex downstream agents can use to mechanically filter.

| Idiom | Regex | What it does |
|---|---|---|
| Refcount macros | `__Pyx_X?(INC\|DEC)REF\b` | INCREF/DECREF/XINCREF/XDECREF wrappers |
| Ownership tracking | `__Pyx_X?(GIVE\|GOT)REF\b` | RefNanny ownership-flow tracking |
| RefNanny lifecycle | `__Pyx_RefNanny(SetupContext\|FinishContext)\b` | Per-function refcount auditing scaffold |
| Error fetch/restore | `__Pyx_(ErrFetch\|ErrRestore\|ErrFetchInState\|ErrRestoreInState)\b` | Exception state save/restore (used in finally blocks, exception chaining) |
| Exception inspection | `__Pyx_(ExceptionMatches\|GetException\|PyErr_GivenExceptionMatches)\b` | Exception type matching |
| Fast call paths | `__Pyx_PyObject_FastCall(Dict\|Method\|Args)?\b` | Cython's vectorcall-fast-path implementations |
| CyFunction internals | `__Pyx_CyFunction_\w+\b` | Cython function objects (closures, decorators, etc.) |
| Memoryview machinery | `__Pyx_memoryview\w+\|__pyx_memoryview\w+\|__Pyx_GetItemInt\w*\b` | Typed-memoryview adapter (`int[:]`, `double[:, :]`, etc.) |
| List/dict optimizations | `__Pyx_PyDict_NewPresized\|__Pyx_PyList_Append\|__Pyx_PyList_Extend\b` | Pre-sized container construction |
| Type checks | `__Pyx_TypeTest\|__Pyx_PyObject_IsTrue\b` | Type validation in cdef-class methods |
| Module init | `__Pyx_check_single_interpreter\|__pyx_pymod_create\|__pyx_pymod_exec_\w+\b` | Multi-phase init machinery |
| Private CPython API | `_PyObject_(GetDictPtr\|Vectorcall)\|_Py_(IsOwnedByCurrentThread\|HashDouble)\b` | Cython runtime uses CPython private APIs; this is upstream Cython's responsibility, not the user's |
| Macros for buffer slots | `__Pyx_(SetItemInt\|GetItemInt\|TypeTest)\b` | Memoryview/list/tuple/dict fast macros |

**Auditor instruction**: any finding that exclusively involves these patterns should be classified ACCEPTABLE (Cython runtime). Only escalate if a maintainer-authored pattern is mixed in.

## 6. Cython-specific bug patterns

The following are real bug classes the AST scripts target. The mapper agent should:
- Reference each Q1/Q2/Q3 finding by ID
- For each, decide HIGH-impact (real bug, FIX) vs LOW-impact (rare/benign) using the project's structural patterns
- Add project-specific reference templates where they exist

### 6.1 Silent noexcept on `cdef int` (Q1)

**The headline Cython 3 pitfall.** Cython 3.0 changed the default exception handling for `cdef`/`cpdef` functions returning C numeric types: without an explicit `except` clause, the function is treated as `noexcept` — Python exceptions raised inside are **silently swallowed at the C-callback boundary**.

**Three valid clauses**:
- `except <value>` — the function returns `<value>` to signal an exception. E.g., `except -1`, `except NULL`.
- `except? <value>` — `<value>` MIGHT mean exception (caller must check `PyErr_Occurred()`).
- `except *` — always check via `PyErr_Occurred()`. For functions where ANY return value is valid.

**Triage by registration context**:

| Function role | Silent-noexcept impact | Triage |
|---|---|---|
| Registered as a C-library callback (e.g., `blosc2_register_codec`, libuv `uv_read_cb`, etc.) | **HIGH** — exceptions silently swallowed at the C boundary; caller never knows. Maps to all 15 blosc2 Q1 findings. | **FIX HIGH** |
| `cdef class` method called only from Python code that checks exception state | **LOW** — Python caller will see the exception via `PyErr_Occurred()` indirectly. Maps to most uvloop Q1 findings. | **CONSIDER LOW** unless caller-side context shows otherwise |
| Internal helper called from another `cdef` function that does check return value | **LOW** | **CONSIDER LOW** |
| Abstract method (`raise NotImplementedError`) | **MEDIUM** — silently masks "abstract not implemented" → caller gets `0` / `False` instead of a crash | **CONSIDER MEDIUM** |
| Debug-only-raise (raises only `if UVLOOP_DEBUG:` or similar) | **LOW** | **CONSIDER LOW** |

**Mapper directive**: for each Q1 finding, classify by registration context. **Do not propagate Q1's HIGH default classification without context-aware triage.** Use grep to find the function's caller(s) before deciding.

### 6.2 `cdef class` `__init__` should be `__cinit__` (Q4)

Cython supports both `__init__` (Python-level constructor; can be called twice) and `__cinit__` (C-level constructor; called exactly once during object construction).

**The reinit double-free pattern**:
- `cdef class` allocates a non-PyObject C resource (malloc, blosc2_*_new, b2nd_*_new, libuv handle init) in `__init__`
- Stores the pointer in a `cdef <T>*` attribute
- `__dealloc__` frees it
- A second `__init__` call (legal in Python) overwrites the pointer without freeing the previous resource → leak; or worse, if the new pointer aliases another live object's pointer, double-free.

**Reference live-reproduced bug**: blosc2 F3 (`SChunk.__init__` re-call leaks `blosc2_schunk *`) and F4 (`NDArray.__init__` re-call double-frees `b2nd_array_t *`).

**Mapper directive**: read the Q4 (`scan_cython_cinit_candidates.py`) output and for each finding:
1. Verify the field is a raw C pointer / handle (`cdef <T>*`, `cdef Foo handle`) — Cython auto-DECREFs Python `object` fields, so those are benign
2. Confirm `__dealloc__` frees the field (otherwise the bug is symmetric: the maintainer never owned a clean lifecycle)
3. Confirm `__init__`'s reassignment isn't preceded by a free/decref of the old value
4. If field is raw + `__dealloc__` frees + no free-before-reassign → **FIX HIGH** (live reinit-leak / double-free)
5. If field is Python `object` → **ACCEPTABLE** (Cython auto-DECREFs)
6. If `__init__` is empty / no-op → **ACCEPTABLE** (defensive Python-only constructor)

The script emits HIGH when RHS is a function call (likely allocation) and MEDIUM otherwise; the mapper's job is to confirm or downgrade based on field type + `__dealloc__` correlation.

### 6.3 Buffer-protocol pairing (Q2)

`PyObject_GetBuffer(obj, &view, ...)` must be paired with `PyBuffer_Release(&view)` on every exit path. Cython does NOT auto-generate this pairing — the maintainer must use `try: ... finally: PyBuffer_Release(&view)`.

**Three structural patterns** (mapper must distinguish):

| Pattern | Description | Q2 reports | Triage |
|---|---|---|---|
| Same-function pairing | GetBuffer + Release in the same function body | Q2 detects correctly; reference template | ACCEPTABLE if in `try/finally`; FIX if not |
| **RAII context object** | GetBuffer in factory `new()`; Release in object's `__dealloc__` or cleanup method | **Q2 false-positives** | **ACCEPTABLE — verify by reading the cdef class** |
| Explicit transfer to libuv/C-callback | GetBuffer pinned for callback duration; Release in completion callback | **Q2 false-positives** | **ACCEPTABLE — verify by tracing the callback registration** |

**Mapper directive**: for each Q2 finding:
1. Find the enclosing function — is it a `cdef class` factory method (e.g., `new()`, `__init__` of cdef class)?
2. If yes, examine the cdef class's `__dealloc__` and any `free_*`/`close()`/`cleanup()` methods for matching `PyBuffer_Release`
3. If found → ACCEPTABLE (RAII pattern)
4. If not found → FIX (real leak) or trace to callback registration if applicable

### 6.4 PyCapsule NULL destructor (Q3)

`PyCapsule_New(ptr, name, NULL)` creates a capsule that performs no cleanup when GC'd — caller is responsible. Multi-wrap (creating two NDArrays from one b2nd_array_t* via capsules) can produce double-free.

**Reference live-reproduced bug**: blosc2 F11 (`as_ffi_ptr` / `array_from_ffi_ptr` ASAN double-free).

**Triage by intent**:

| Intent | Example | Triage |
|---|---|---|
| Internal lifecycle handoff | blosc2's NDArray construction sites | CONSIDER (audit consumers) |
| Documented FFI bridge | uvloop's `libuv_get_loop_t_ptr` | ACCEPTABLE (intentional design) |
| Multi-wrap of same pointer | F11 pattern | **FIX HIGH** (real double-free) |

**Mapper directive**: for each Q3 finding:
1. Read the function's docstring / comments — is it documented as an FFI bridge?
2. Trace the capsule's consumers — are they unique-ownership or multi-wrap?
3. Classify accordingly

### 6.5 `nogil` scopes touching Python state (Q5)

Cython auto-acquires the GIL for some operations inside `with nogil:` (e.g., `__Pyx_AddTraceback`, `__Pyx_Raise` are wrapped). But **error paths sometimes miss this**, and bare `raise` / `print` / f-strings inside `cdef ... nogil:` bodies are real bugs.

**Reference live-reproduced bug**: blosc2 F14 (`matmul_prefilter` calls `__Pyx_AddTraceback` without GIL on dtype-error path).

**Mapper directive**: read the Q5 (`scan_cython_nogil_pyobject.py`) output and for each finding:
1. Confirm scope — is the candidate truly inside an unprotected nogil region? Q5 already walks ancestors, but verify by reading source (Cython sometimes implicitly wraps with-gil around `raise` if `except *` is set; the script does not detect this).
2. If kind = `raise` and no enclosing `with gil:` and no `except *` declaration → **FIX HIGH**
3. If kind = `print` / f-string / comprehension and no enclosing `with gil:` → **FIX HIGH** (these have no implicit-gil wrapping)
4. If the surrounding cdef function is declared `with gil` (not `nogil`) → script-misclassification; **ACCEPTABLE**
5. Cross-reference with §7's `freethreading_compatible` directive: a project that opts into FT must be especially strict about this — `Py_MOD_GIL_NOT_USED` makes nogil-violation crashes more likely under contention.

### 6.6 Module-level free-threading status

Cython 3.x default: `Py_MOD_GIL_USED` (modules import in GIL-re-enabled compatibility mode under 3.13t/3.14t).

To opt into free-threading: `# cython: freethreading_compatible=True` directive. This makes Cython emit `Py_MOD_GIL_NOT_USED`.

**Mapper directive**: report the status:
- `Py_MOD_GIL_USED` (default) — note as conservative; agents should verify via `PyUnstable_Module_SetGIL` calls in generated `.c`
- `Py_MOD_GIL_NOT_USED` (claimed) — escalate priority on FT-relevant findings (atomics, lock discipline)

### 6.7 `__dealloc__` calling code that holds GIL through expensive C cleanup

If a `cdef class`'s `__dealloc__` calls a C-side cleanup function that's slow (e.g., `b2nd_free` traversing a large data structure), and the call is made while holding the GIL, Python 3.14 mass-GC will hang because no other Python thread can run during the dealloc.

**Reference live-reproduced bug**: blosc2 F12 (`NDArray.__dealloc__` holds GIL through `b2nd_free`).

**Mapper directive**: for each `cdef class` `__dealloc__`:
1. Check whether it calls a C cleanup function
2. If yes, check whether the call is wrapped in `with nogil:`
3. If not, flag as a 3.14-mass-GC-hang candidate

## 7. Project-specific structural patterns

These are NOT detectable by the AST scripts. The mapper agent must surface them by reading source code. Examples from the calibration corpus:

| Project | Pattern | Where | Why it matters for triage |
|---|---|---|---|
| blosc2 | Reference template — buffer protocol | `vlcompress` blosc2_ext.pyx:1289-1355 | Correct `try/finally PyBuffer_Release`; compare Q2 findings against this |
| uvloop | RAII context object | `_StreamWriteContext` (stream.pyx), `_UDPSendContext` (udp.pyx) | GetBuffer in `new()`, Release in `__dealloc__`/`free_bufs()` — Q2 false positives |
| uvloop | libuv callback registration shape | `cdef void X(uv_handle_t*) noexcept with gil` | The actual callback boundary; Q1's `cdef int` candidates are NOT here |
| blosc2 | C-Blosc2 callback registration | `cdef int general_<role>(blosc2_<role>_params *params)` | The actual Q1-relevant boundary |
| h3-py | `cpdef (T1, T2) func()` tuple-return semantics | h3-py/_cy/util.pyx, vertex.pyx | Cython's tuple-typed returns; downstream agents may misread these |

**Mapper directive**: apply best-effort discovery — read the largest .pyx files, examine `cdef class` definitions, look for reference templates and recurring structural shapes. Surface 1-2 patterns where they exist. If a project genuinely has no project-specific patterns (small / generic / template-driven), state that explicitly: "No project-specific structural patterns identified; downstream agents should rely on generic Cython triage in §6." Do not invent patterns to fill the section.

## 8. What this playbook does NOT cover

Patterns the mapper agent and AST scripts cannot detect:
- Refcount semantics across function boundaries (cross-file dataflow)
- Lock-discipline correctness when locks are passed as cdef-class members
- Race conditions between threads operating on shared cdef-class state
- Logical bugs in the maintainer's algorithm
- Whether a `cdef class` is intended to be subclassed
- Whether a `with nogil:` block is correctly assuming the calling thread holds the GIL

These remain the domain of the existing audit agents (refcount-auditor, error-path-analyzer, gil-discipline-checker, etc.). The mapper provides orientation; the audit agents do the deep work.

## 9. Cython version differences

| Version | Change | Impact on triage |
|---|---|---|
| 3.0 | Default `noexcept` for `cdef int` without `except` clause | Q1 detection becomes meaningful; pre-3.0 code may have implicit-except-* |
| 3.1 | `freethreading_compatible` directive added | Free-threading orientation possible |
| 3.2 | `__Pyx_FREETHREADING_COMPATIBLE` macro added | Generated `.c` includes the FT branch unconditionally |
| 3.3+ | (TBD as releases land) | Update this playbook |
| 4.0 (future) | Removal of `DEF` directive (and other deprecations) | Build-system finding: pin `cython<4` or migrate `DEF` → `cdef enum` |

**Mapper directive**: identify the actual Cython version used (per §4) and document version-specific concerns. **If the version cannot be determined, default to Cython 3.2 and note the assumption explicitly in the orientation guide.**
