# cext-review-toolkit Improvement Plan

Based on analyzing ~15 C extensions: cffi, scipy, pandas, multidict, aiohttp, Cython runtime, mypyc runtime, cereggii, charset-normalizer, matplotlib, pyhacl, h5py, numpy, lxml, memray.

## Priority 1: Auto-detect code generation in discover_extension.py (Small effort, High impact)

Add a `code_generation` field to extension discovery output: `"hand_written"`, `"cython"`, `"mypyc"`, `"pybind11"`, `"mixed"`.

Detection heuristics:
- Cython: `"Generated by Cython"` in first 200 bytes of .c files, or .pyx files present
- mypyc: `CPyDef_` / `CPyStatic_` / `CPyModule_` patterns, or `mypyc` in header
- pybind11: `PYBIND11_MODULE` or `py::class_` in .cpp files
- Hand-written: none of the above

The explore command should use this to:
- Skip refcount/error-path/null scanners for Cython/mypyc (95-100% FP rate)
- Focus agents on .pyx/.py source instead of generated .c
- Skip module-state/stable-abi/version-compat for generated code (answers are always "Cython/mypyc limitation")
- Tell agents what kind of code they're looking at so they can adjust expectations

Implementation: ~50 lines added to discover_extension.py, ~20 lines in explore command.

## Priority 2: Fix type_spec_missing_sentinel to accept {0, 0} (Trivial, Medium impact)

In scan_type_slots.py, the sentinel check pattern-matches for `NULL` specifically. Change to also accept `0` as the second element. Every Cython extension generates `{0, 0}` — this eliminates false positives across all Cython/mypyc projects.

Implementation: ~5 lines changed in scan_type_slots.py, update test.

## Priority 3: Exclude CPython type slots from callback_without_gil (Small, Medium impact)

In scan_gil_usage.py `_check_callback_without_gil`, add an allowlist for functions assigned to known CPython type slots. Functions whose names appear in struct initializer contexts like `.tp_dealloc = funcname` should not be flagged. Slot names to exclude: tp_dealloc, tp_traverse, tp_clear, tp_init, tp_new, tp_repr, tp_hash, tp_call, tp_str, tp_getattro, tp_setattro, tp_richcompare, tp_iter, tp_iternext, nb_*, sq_*, mp_*, bf_*.

This was suggested by the charset-normalizer GIL agent and would eliminate ~50% of GIL false positives.

Implementation: ~30 lines in scan_gil_usage.py, update test.

## Priority 4: New scanner — PyErr_Clear audit (Small effort, High impact)

New script: `scan_pyerr_clear.py`

Finds all `PyErr_Clear()` calls and checks if they're preceded by `PyErr_ExceptionMatches()`. Reports unguarded clears as findings.

This was the **dominant bug class in the Cython runtime** (~25 sites swallowing MemoryError) and also appeared in cffi, pandas (ujson), scipy, and mypyc runtime. Pattern:
- GOOD: `if (PyErr_ExceptionMatches(PyExc_KeyError)) { PyErr_Clear(); }`
- BAD: `PyErr_Clear();` (swallows MemoryError, KeyboardInterrupt, SystemError)

Finding types:
- `unguarded_pyerr_clear`: PyErr_Clear without preceding ExceptionMatches check
- `broad_pyerr_clear_in_hot_path`: unguarded clear in a function called frequently

Implementation: ~200 lines. Uses tree-sitter to find PyErr_Clear calls, then checks the preceding statements/conditions for PyErr_ExceptionMatches.

## Priority 5: New scanner — Resource lifecycle (Medium effort, High impact)

New script: `scan_resource_lifecycle.py`

Tracks allocation → free pairing for non-PyObject resources:
- `malloc`/`calloc`/`realloc` → `free` (found bugs in scipy, h5py, pandas)
- `PyMem_Malloc`/`PyMem_Calloc` → `PyMem_Free`
- `H5Tcreate`/`H5Tcopy`/`H5Tget_member_type` → `H5Tclose` (h5py bugs)
- `H5Screate_simple`/`H5Dget_space` → `H5Sclose` (h5py bugs)
- `PyObject_GetBuffer` → `PyBuffer_Release` (aiohttp bug)

Reports allocations that reach a function exit (including error paths via goto) without their matching free.

This catches the most impactful bug class we found: resource leaks on error paths. The h5py analysis found 5 HDF5 handle leaks, scipy had 6+ malloc leaks, pandas had the use-after-free in int64ToIso.

Implementation: ~400 lines. Uses tree-sitter to build a simple flow graph, tracks allocation/free calls through the function, reports unmatched allocations on any exit path.

The scanner would need a configurable "resource pairs" file (like api_tables.json) so users can add project-specific pairs (e.g., HDF5 functions for h5py).

## Priority 6: Cython-aware NULL check patterns (Small, Medium impact)

In scan_null_checks.py, add recognition for Cython's generated NULL check patterns:
- `if (unlikely(!result))` as a NULL check
- `__PYX_ERR(file, line, label)` as an error handler (implies preceding check)
- `if (unlikely(result == ((type)NULL)))` as a NULL check

This would reduce NULL scanner false positives from 95-100% to ~20-30% on Cython code.

Implementation: ~30 lines in scan_null_checks.py.

## Priority 7: C/Python parity agent (Medium effort, Medium impact)

New qualitative agent: `parity-checker.md`

For extensions that ship both a C/Cython fast path and a Python fallback:
- Identify dual implementations (C parser + Python parser, C encoder + Python encoder)
- Compare validation logic (what inputs are rejected by each)
- Compare error handling (what exceptions are raised)
- Flag behavioral differences as potential security issues

This found the most security-relevant bugs in aiohttp (TE+CL conflict, CTL char validation, URL scheme validation, upgrade protocol set).

Agent would be in Group B (after architecture context) since it needs to understand the module structure.

Implementation: Agent prompt ~200 lines. No script needed — agent uses Grep/Read.

## Priority 8: New agent — test-investigation-agent for cext-review-toolkit

Port the test-investigation-agent from code-review-toolkit to cext-review-toolkit, with C-extension-specific enhancements:
- Extract `PyMethodDef` tables from C code to map Python test calls to C functions
- Understand Cython .pxd declarations for test-to-source mapping
- Focus on HDF5/NumPy/etc. resource lifecycle invariants (not just Python refcounts)

Lower priority because the Python test-investigation-agent already works well for C extensions when given context (as proven with multidict).

## Summary of Bug Classes Found

| Bug Class | Extensions Affected | Proposed Fix |
|-----------|-------------------|--------------|
| Unconditional PyErr_Clear | Cython, cffi, pandas, scipy, mypyc | Priority 4 (new scanner) |
| Resource leaks on error paths | h5py, scipy, pandas, multidict | Priority 5 (new scanner) |
| Heap type missing Py_DECREF(tp) | mypyc, multidict (partial) | Existing type-slot scanner catches this |
| C/Python behavioral parity gaps | aiohttp | Priority 7 (new agent) |
| Scanner FP on generated code | All Cython/mypyc extensions | Priorities 1-3, 6 |
| Missing tp_clear slot registration | Cython runtime, mypyc | Existing type-slot scanner catches this |
| Complex `real + imag * I` | scipy | Too specialized for a general scanner |
| Uninitialized exception objects | scipy (ARPACK) | Existing module-state scanner should catch this |

## Implementation Order

1. Priorities 1-3 (scanner FP reduction) — immediate, small effort, high quality-of-life impact
2. Priority 4 (PyErr_Clear scanner) — next, small effort, catches a whole new bug class
3. Priority 5 (resource lifecycle) — medium effort but highest bug-finding potential
4. Priorities 6-8 — follow-up improvements
